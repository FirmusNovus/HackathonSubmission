# Phase 0 Research — Lex Nova MVP

This document captures the technical decisions that fall *inside* the user-pinned tech envelope (Next.js + SQLite + shadcn + Solidity + Anvil + testnet path). It also records what was rejected and why, so the plan can be replayed.

## Decision 1: Wallet integration protocols

**Decision**: OID4VCI for credential issuance (pre-authorized code grant + DPoP), OID4VP with DCQL for presentation, SD-JWT VC (`vc+sd-jwt`) as the credential format, x509_san_dns Draft-23 syntax for the verifier `client_id`, and signed `request_object` JWS with embedded `x5c`.

**Rationale**: All four are already validated end-to-end against a real wwWallet by the spike at [../../docs/spike/wallet-integration/](../../docs/spike/wallet-integration/). Switching to anything else would invalidate the spike's hard-won quirks list. The constitution's principle IV (Standards-Compliance Over Novelty) and principle V (Spike-Validated Before Specced) both point here.

**Alternatives considered**:
- `presentation_definition` (OID4VP-classic) — wwWallet returns `MISSING_DCQL_QUERY`. Rejected.
- mDOC / mDL credential formats — out of scope; SD-JWT VC matches EUDI ARF and is what wwWallet emits.
- Issuing without DPoP — pre-auth grant flow in wwWallet expects DPoP. Rejected.

**Validated wwWallet constraints to retain**:
- Issuer + verifier metadata MUST include `Cache-Control: no-store` (wwWallet caches metadata for 30 days otherwise).
- The `vp_token` returned from a presentation is a JSON-stringified object whose value can be string OR array depending on `wallet-common` version — code MUST handle both.
- The `iss` claim in issued credentials MUST be an HTTPS URL, not the issuer's `did:key`.
- `client_id` for the verifier uses `x509_san_dns:<hostname>` (Draft-23). Without the prefix, presentations fail.

## Decision 2: End-to-end-encrypted messaging substrate

**Decision**: WebCrypto in the browser. ECDH P-256 between the two parties' wallet keys derives a shared secret, HKDF-SHA-256 derives per-message AES-GCM keys, ECDSA P-256 signs each message envelope. Ciphertext blobs are stored in SQLite as opaque bytes; the server has no key path to plaintext. The transport is HTTP POST/GET against the Next.js API for the MVP — XMTP / Waku is documented production trajectory only.

**Rationale**: WebCrypto is browser-native (no extra deps, audited by the platform), and the constitution's invariant 1 forbids any server-side decryption capability. Storing ciphertext in the same SQLite file as everything else is the simplest substrate that keeps the platform's promise of "subpoena returns ciphertext."

**Alternatives considered**:
- libsodium / `tweetnacl` — adds a JS dep with no win over WebCrypto on modern browsers.
- XMTP integration — meaningful production substrate but multi-day dev cost and no privilege improvement over the local stub for the demo. Production trajectory.
- Storing messages on chain (in calldata) — gas explodes; pointless when the server can store ciphertext at zero risk to the privilege boundary.

## Decision 3: Per-engagement transcript anchoring

**Decision**: Each message hash (SHA-256 of the signed envelope) is appended as a leaf in a per-engagement incremental Merkle tree (depth 16 supports 65k messages, more than enough). The Merkle root is committed on chain via `LegalEngagementEscrow.anchorTranscript(engagementId, root)` at every milestone fund / deliver / release / dispute / claim / resolve / close event. Between events, the off-chain root is the latest leaf-derived root; on-chain commits make ranges of history tamper-evident.

**Rationale**: Constitution invariant 5 requires per-engagement tamper-evidence anchored on chain. Anchoring at every milestone state transition gives the demo natural beats to show "this is now immutable." Using SHA-256 instead of Poseidon keeps the off-chain math cheap; we don't need ZK over the transcript.

**Alternatives considered**:
- Anchor every message — too many transactions, demo-disrupting gas.
- Anchor only at engagement close — leaves long mid-engagement windows unanchored; weakens the audit story.
- Use Poseidon for compatibility with the conflict-of-interest circuit — overkill, the transcript and the conflict circuit don't share inputs.

## Decision 4: Conflict-of-interest ZK toolchain

**Decision**: Noir 1.0.0-beta.20+ for circuit authoring, UltraHonk (Aztec barretenberg) as the proving backend. The circuit proves non-membership of a Pedersen-hashed pseudonymous client identifier in a fixed-size-N (N=8 for MVP) array of Pedersen-hashed commitments published by the lawyer. Verifying key is hardcoded into a Solidity verifier contract auto-generated by `bb`. Proof generation runs in the browser via `@aztec/bb.js`; verification runs on chain at engagement-creation time.

**Rationale**: Noir is the most ergonomic option as of 2026 and the constitution names it as the validated choice. UltraHonk verifier sizes fit comfortably under L2 calldata limits. N=8 is enough for a demo lawyer to plausibly have "current clients" without breaking enumerability.

**Alternatives considered**:
- Circom + Groth16 — older, requires trusted setup, more JS boilerplate. Rejected.
- snarkjs PLONK — comparable to UltraHonk but worse DX for small circuits.
- Off-chain conflict check — undermines the credibility of the privacy story.

## Decision 5: Smart-contract design

**Decision**:
- `LegalEngagementEscrow.sol` holds milestone state and is the trust anchor for the asymmetric mechanism. It reads capability attestations from EAS and from a thin `AttestationManager.sol` wrapper that exposes a stable `hasCapability(address, bytes32)` function so the gating layer doesn't have to talk to the EAS schema registry directly.
- `AttestationManager.sol` registers three EAS schemas (`verified_lawyer`, `verified_client`, `verified_arbiter`) at deploy time and exposes write functions gated by the operator address.
- `IZKConflictVerifier.sol` is the auto-generated Noir verifier interface; `LegalEngagementEscrow.fundFirstMilestone` requires a successful proof.
- All cooldown and capability checks are in modifiers on `LegalEngagementEscrow` itself — never enforced off-chain (constitution invariant 6).
- Milestone funds denominated in the chain's native asset (ETH on the demo chain) per the spec's clarification. The contract surface is shaped so a future ERC-20 variant deploys alongside without changing the API visible to clients/lawyers/arbiters (a sibling `LegalEngagementEscrowERC20.sol` is the documented production trajectory).

**Rationale**: Three small contracts with clear seams. Foundry tests cover the asymmetric invariants explicitly. EAS gives the on-chain handshake (constitution invariant 2) and is already deployed on Base Sepolia, removing one chunk of deployment work for testnet path.

**Alternatives considered**:
- Single monolithic contract — harder to test, harder to upgrade just one piece.
- Diamond / proxy pattern — overkill for a hackathon.
- ERC-20 stablecoin from day one — adds approve flow + token contract for zero MVP value.

## Decision 6: Contract version pins

**Decision**: Solidity 0.8.28, OpenZeppelin Contracts v5.2.0 specifically.

**Rationale**: Constitution memory and prior spike work pin OZ v5.2.0; later patch versions have changed access control surfaces. Solidity 0.8.28 is the latest stable at planning time and includes the fixes we need around custom errors.

## Decision 7: Local chain + testnet path

**Decision**:
- Local development: Anvil with the default 10 funded accounts. `evm_increaseTime` available for the cooldown demo beat.
- Testnet path: Base Sepolia. EAS is already deployed at well-known addresses there; gas is trivial; chain is fast.
- Deployment is a `forge script Deploy.s.sol --rpc-url $RPC --broadcast` away from working on either. The only application-side change is `process.env.NEXT_PUBLIC_RPC_URL` and the deployed-contract address constants.

**Rationale**: Anvil is what the user named explicitly. Base Sepolia is the cheapest L2 testnet with EAS already deployed; using it as the production-trajectory target avoids reinventing schema deployment. Constitution principle IV demands standards compliance; EAS-on-Base-Sepolia is the standard path.

**Alternatives considered**:
- Sepolia (L1) — gas higher, slower, no native L2 story for production.
- Optimism Sepolia — EAS deployed, equivalent to Base. Either works; pick Base because the wider EUDI/EAS demos in 2026 trend Base-first.
- Hardhat instead of Anvil — slower, the user pinned Anvil.

## Decision 8: Auth, persona staging, and the operator role

**Decision**:
- SIWE (EIP-4361) for app auth. After credential presentation, the SIWE-bound address is the user's identity; the credential never re-presents.
- 10 anvil accounts get *issuer-side knowledge* registered at boot via two independent seed steps — one per issuer process — both run by `pnpm scripts:seed`: the bar-issuer's seed populates its `subjects` table with the lawyer roster (anvil indices 1–5) and ensures its own ES256 signing JWK exists; the pid-issuer's seed does the same for the lawyer + client roster (indices 1–6) with the EUDI ARF claim shape. Each seed writes only into its own service's data dir (`apps/{bar,pid}-issuer/data/`); neither touches the platform's DB. **No on-chain attestations and no `verified_users` rows are written by the seeds.** The platform starts empty. Each persona must complete the real OID4VP onboarding flow on stage to land on the platform's verified-user surface and on chain.
- Account allocation: account 0 = platform operator; accounts 1–5 = lawyers (Anna, Carlos, Dieter, Sofia, Eva); account 6 = client (Marta); accounts 7–9 reserved for second client / multi-engagement scenarios. Eva's additional `verified_arbiter` capability is granted by the operator (on stage, via the operator capability admin page in Phase 7) after she completes lawyer onboarding.
- The operator's private key is loaded from `.env.local` (never committed). The operator can revoke any capability and grant `verified_arbiter` to a wallet that already holds `verified_lawyer`. The operator MUST NOT be able to grant `verified_lawyer` or `verified_client` directly — those originate only from completed credential presentations (FR-007).

**Rationale**: SIWE is the constitution's named auth standard. Pre-staging is the simplest path to a demo where personas all have credentials before the stage starts; the capability admin page is what shows on stage that "the operator can fix mistakes but can't create lawyers."

**Alternatives considered**:
- WalletConnect modal for arbitrary wallets — fine for production, friction for a demo.
- OAuth proxy — sidesteps SIWE, breaks the principle.

## Decision 9: Explicitly out of scope (slide-only)

The constitution's Demo and Production Discipline section enumerates these; they are repeated here so Phase 1 design doesn't accidentally pull them back in:

- TIR (Trusted Issuers Registry) runtime lookup — operator review at attestation time satisfies the equivalent gate for the demo. Production trajectory.
- Threshold cryptography for distributed identity escrow — Tier 3.5; out.
- Multi-sig arbiter committees — first-claim model with a single arbiter (Eva, promoted by the operator after she completes lawyer onboarding) suffices for MVP; the contract semantics already permit a multi-arbiter pool. Production trajectory.
- ERC-5564 stealth addresses for per-engagement client unlinkability — OID4VCI batch unlinkability covers the audience-facing claim.
- QES (Qualified Electronic Signatures) via QTSP partner — paperwork-gated, multi-week. Out.
- Full XMTP / Waku messaging transport — local stub is the same cryptographic shape; transport is a swap, not a protocol change.
- Identity unsealing for fraud / regulatory escalation — explicitly NOT IMPLEMENTED in MVP per spec assumption.

## Decision 10: Demo time-skip mechanism

**Decision**: For the lawyer-cooldown beat, the demo issues `cast rpc evm_increaseTime <secs> && cast rpc evm_mine` against the running anvil to skip past 30 days. The Next.js app reads `block.timestamp` via viem and shows the live "escalation now possible" state immediately.

**Rationale**: Spec assumption explicitly allows a time-skip mechanism. Anvil exposes the standard Hardhat-compatible `evm_increaseTime`; viem polls `eth_blockNumber` so the UI updates without manual refresh.

**Alternatives considered**: Make the cooldown configurable to a few seconds for the demo — rejected because then the asymmetric mechanism's *meaning* changes on stage, and the constitution's Honest-Framing principle prefers visible time-skip over "the cooldown is 5 seconds."
